中央 17 点を 90 点超へ — 業界トップを取りに行く改善 10 ステップ

2026 年 5 月 15 日 公開 / 最終更新 2026 年 5 月 15 日 / 読了目安 11 分

この記事の結論

東証プライム上場 1,553 社の WEB 品質中央値は 17 点（100 点満点）、最高 52 点で ★ 認定取得は 0 社。つまり「業界トップでも 50 点台が天井」が現状です。これは中小企業にとって、基準を理解し正しく実装すれば業界 1 位の WEB 品質を取りに行ける余白がある証明でもあります。本記事は 17 点を 90 点超へ持ち上げる実装 10 ステップを、優先順位・所要工数・依存関係付きで提示します。

この 10 ステップは誰のためのものか

  • □ 自社サイトを業界平均（17 点）以上に磨きたい中小企業の経営者・WEB 担当者
  • □ 「★ 認定取得 0」の業界天井を超えてブランド差別化したい事業者
  • □ AI 検索（ChatGPT / Perplexity / Google AI Overviews）への引用を本気で取りに行く事業者
  • □ 制作会社に依頼する前に、要件定義レベルで品質基準を握っておきたい発注者

全体マップ：10 ステップと配点

本ロードマップは「中央 17 点」のサイトに対し、10 ステップ全て実施することで +83 点（理論最大）を積み上げ、合計 90 点超に到達する設計です。実装順は 「致命的 → 計測可能 → 機械可読 → AI 適合」の優先順位。総所要工数は 30-50 日（実装者 1 名フルコミット換算）。

Step 1. 致命的 NG の即時解消（+15 点 / 1-3 日）

東証プライム上場 1,553 社中 9.6%（149 社）が引っかかった「致命的 NG」を最優先で潰します。Phase E 内訳：

・WordPress 管理面露出（129 社 / NG の 87%）— /wp-admin/ パス変更プラグイン or .htaccess で IP 制限

・CMS バージョン情報露出（14 社）— Functions.php で wp_generator 削除 / HTTP ヘッダ X-Powered-By 削除

・HTTPS 非対応（13 社）— Let's Encrypt or 商用証明書で常時 HTTPS 化 + HSTS

・SSL 証明書問題（5 社）— 自動更新設定 + ドメイン一致確認

「自社サイトはどうか」は securityheaders.com や SSL Labs Server Test で即時無料チェック可能です。

Step 2. Core Web Vitals 全 GOOD（+15 点 / 5-10 日）

Google 公式の Core Web Vitals 3 指標を全て GOOD（緑）に：

・LCP（最大コンテンツ描画）: 2.5 秒以内 — Hero 画像 preload / WebP 化 / Critical CSS インライン

・INP（次のペイントまでの相互作用）: 200ms 以内 — 重い JS の defer / 長時間タスクの分割 / メインスレッド ブロック削減

・CLS（累積レイアウトシフト）: 0.1 以内 — 画像 width/height 属性必須 / フォントの font-display:swap + size-adjust

計測は PageSpeed Insights（実測 CrUX フィールドデータ）と Chrome DevTools Performance タブで。詳細は Core Web Vitals 解説記事を参照。

Step 3. 構造化データ JSON-LD 完備（+10 点 / 3-5 日）

機械が内容を正確に理解する土台。最低 5 種類を JSON-LD で実装：

・Organization — 全ページ共通の組織情報 / @id でグローバル参照

・BreadcrumbList — パンくず構造 / 階層 SEO に必須

・Article — 記事ページ / headline / datePublished / author

・FAQPage — Q&A 形式コンテンツ / リッチリザルト対象

・LocalBusiness（該当時） — 地域事業者 / address / areaServed

検証は Google リッチリザルトテスト と Schema.org Validator。詳細は JSON-LD 実装ガイド。

Step 4. WCAG 2.2 AA 準拠（+10 点 / 5-10 日）

W3C 2023 年 10 月勧告の最新基準。AA レベルで必須化された主要 4 項目：

・タップターゲット最低 24×24px（実装上は 44×44px 推奨）

・コントラスト 4.5:1 以上（大きな文字は 3:1） — axe DevTools 等で機械検証可

・ランドマーク要素 — header / nav / main / aside / footer の構造化

・aria 属性整備 — aria-label / aria-labelledby / aria-current 等

計測は axe DevTools・Lighthouse Accessibility・Pa11y。詳細は WCAG 2.2 解説記事。

Step 5. 厳格 CSP + Secure Headers（+8 点 / 2-3 日）

OWASP Secure Headers Project の主要 5 つを設置：

・Content-Security-Policy — XSS 構造的遮断 / require-trusted-types-for 'script' で Trusted Types 強制

・Strict-Transport-Security — HSTS Preload 登録（hstspreload.org）

・X-Frame-Options: DENY — クリックジャッキング防止

・X-Content-Type-Options: nosniff — MIME スニッフィング防止

・Referrer-Policy: strict-origin-when-cross-origin — 適切な Referrer 制御

計測は securityheaders.com で A+ ランク取得を目標。詳細は セキュリティ 5 原則記事。

Step 6. AI 検索適合（LLMO）（+10 点 / 継続）

ChatGPT / Perplexity / Google AI Overviews に引用される文章設計。KDD 2024 論文（GEO: Generative Engine Optimization）で実証された手法：

・冒頭直答 — 40-60 語で問いに直答（曖昧な前置きを排除）

・3 文段落 — 段落を短く保ち、AI が抜き出しやすい単位に

・統計 + 一次出典 — 150-200 語ごとに数値 + URL（権威性担保）

・代名詞回避 — 主語の繰り返しで段落単位の独立性を確保

これらは「魔法の道具」ではありません。Pew Research（2025-07）で AI 要約時の検索クリック率は 8%（非要約 15%）に下がっており、AI 引用に入らないと存在しないのと同じです。詳細は LLMO 解説記事。

Step 7. 画像・フォント最適化（+5 点 / 2-3 日）

Step 2 の Core Web Vitals 改善と連動する細部最適化：

・WebP / AVIF 化 — PNG/JPEG 比 30-50% 軽量化 / picture 要素で fallback

・遅延読み込み — loading="lazy" + 適切な width/height で CLS ゼロ維持

・フォントサブセット化 — Noto Sans JP 等を unicode-range で日本語常用漢字に絞る

・font-display:swap + size-adjust — FOIT / FOUT 軽減

詳細は 画像・フォント最適化記事。

Step 8. Wikidata Q コード + sameAs（+3 点 / 1-2 日）

自社・代表者・関連組織を Wikidata エンティティとして整備し、Schema.org の sameAs で連動：

・自社が Wikidata に既に登録されているかを Wikidata 検索で確認

・未登録なら Wikidata 新規ページ作成（業種・所在地・代表者・設立年などを構造化）

・Schema.org Organization の sameAs に Wikidata URL を追加

AI クローラーは Wikidata を「信頼できる事実源」として参照するため、Q コードは AI 検索エンジンとのエンティティ照合に効きます。詳細は Wikidata for AI記事。

Step 9. llms.txt + robots.txt 整備（+2 点 / 半日）

AI クローラー向けの明示的ポリシー設置：

・/robots.txt に GPTBot / ClaudeBot / PerplexityBot / Google-Extended の Allow/Disallow を明示

・/llms.txt にサイト構造の AI 向け索引を設置（実効性は限定的だが害なし）

注意：llms.txt は 2026 年 5 月時点で主要 AI クローラーがほぼ参照していません（Google John Mueller も 2025-06 に「どの AI システムも使っていない」と公言）。設置しても害はありませんが、過剰評価しないこと。詳細は llms.txt 論争記事。

Step 10. 継続検証ループ（+5 点 / 運用フロー）

1-9 で 85 点台に達したサイトを 90 点超で維持し続けるには、月次の継続検証が必要です：

・月次 PageSpeed Insights（実測 CrUX 28 日窓） — Core Web Vitals の劣化を早期検知

・月次 Lighthouse — Performance / Accessibility / Best Practices / SEO の 4 軸を 90+ 維持

・月次 HARTON 機械検証 — 4 軸 100 点満点の総合スコア推移を追跡

・四半期 出典更新 — 引用している統計・論文・規格の最新版チェック（WCAG 2.3 / Web Vitals 新指標等）

・年次 リサーチ更新 — Verizon DBIR / Web Almanac / KDD 等の年次レポートが出たタイミングで全 Insights 記事を見直し

「制作して終わり」を脱して「育てる」運用に切り替えるのが、90 点超を維持する現実的な方法です。詳細は 公開はゴールではない記事。

このサイト自体が実証です。tcharton.com は spec-checker 内製ツールで 11,000 項目以上を全ページに対し毎回チェックし、pre-push hook で S-RANK（FAIL:0 / 合格率 100%）未満は push を阻止する運用にしています。実装の全コードは GitHub で公開。方法論ページも参照してください。

よくある質問

10 ステップ全てやらないと 90 点に届きませんか？
理論上は Step 1-5 + 10（必須 + 高優先）だけで 70 点台に到達可能です。Step 6 LLMO + 残りで 90 点超を狙います。ただし Step 10 継続検証は必須で、これがないと劣化して維持できません。
どのステップから始めるべきですか？
Step 1 致命的 NG の即時解消が最優先。これは 1-3 日で +15 点が見込め、かつセキュリティリスクを即座に下げます。次は Step 2 Core Web Vitals。Step 3 以降は並行可能ですが、Step 10 継続検証ループの仕組みは早めに整えてください。
既存サイトを 17 点 → 90 点に持ち上げる費用感は？
サイト規模・既存実装の質によりますが、中規模サイト（20-50 ページ）で全 10 ステップ実装は 100-300 万円程度が市場相場の目安です。HARTON は機械検証で進捗を可視化し、ステップ単位の分割発注も承ります。料金ページ参照。
WordPress でも 90 点に届きますか？
届きます。ただしテーマ・プラグインの選定で大きく差が出ます。重いテーマ・プラグイン多用は Core Web Vitals で大きく失点します。プラグインは 5 個以下を原則とし、Step 1 のセキュリティ対策で WP 管理面露出を必ず塞いでください。
★ 認定取得を維持する自信がない場合は？
無理に取得を目指す必要はありません。中央値 17 点を 40-60 点に持ち上げるだけでも業種上位 10% に入れます。「業界トップ」を狙うか「業界上位」で十分かは、自社の競合優位戦略と相談して決めてください。
