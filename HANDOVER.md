## 2A+2B 統合プロジェクト：地域 × 業種 ポートフォリオ LP 構築

### 2.1 目的・背景（v2 統合版）

**v1 旧課題**:
- `cases/` ページは枠組みのみで、実際の納品サイトの**実物展示が不在**
- 「買い切り型 30〜80 万円・S クラス保証」の説得力を視覚的に証明する物件が必要

**v2 統合視点（提案②取込）**:
- 地域中小経営者は「**業種＋地域名＋課題**」のロングテール KW で検索する
- 単純なポートフォリオでは SEO/LLMO 効果が局所的
- **デモサイト = 地域 × 業種 LP** とすることで「実物展示」と「検索対策」を二重達成

**ゴール**:
業種別の本物の S クラスサイトを **地域 × 業種ロングテール LP として再設計**し、`cases/` から実物リンクで展示。**Demo + LP 二重機能**を担う。

### 2.2 対象業種 × 地域（v2 統合版）

| # | 業種 | 地域候補 | 確定例 |
|---|------|---------|--------|
| **A** | **士業（仕業）** | 沼津市 / 三島市 | 例: **沼津市 × 司法書士事務所** |
| **B** | **飲食** | 沼津市 / 三島市 / 富士市 | 例: **三島市 × 和食割烹** |

**業種 + 地域選定はユーザー判断**（着手時確認）。地域名は LP の URL / title / 構造化データに必須反映。

### 2.3 技術要件（必達 / v2 強化版）

- ✅ SPEC v3.2 完全準拠（spec-checker.js で S-RANK 達成）
- ✅ JSON-LD 5 種（業種に応じて `LegalService` / `Restaurant` を主役に）
- ✅ Lead Evidence Block 必須
- ✅ Body Theme Variants（marketing / reading）使い分け
- ✅ CSP / セキュリティヘッダ揃え
- ✅ Cloudflare Web Analytics + GA4（必要に応じて別測定 ID）
- ✅ レスポンシブ完全対応・WCAG 2.2 AA
- ✅ 各業種に最適化された **独自カラーパレット**（tcharton 本体の teal は使わない）
- ✅ 5〜8 ページ程度の小規模だが完結した構成
- ✅ サンプル屋号は架空（実在企業との混同防止のため明確に「サンプル」「デモ」表示）
- 🆕 **地域×業種ロングテール対応**: title / h1 / meta description / canonical / JSON-LD areaServed に**地域名を必須反映**
- 🆕 **独自地域データ最低 2 件埋込**: 「○市製造業の従業員数別 IT 化率」「○市の事業所数推移」等を**経済センサス・市公式統計**から引用（一次ソース URL 明記）
- 🆕 **Doorway Page Penalty 回避**: テンプレ流用禁止。各ページ独自データ・独自課題分析を必須化

### 2.4 推奨ページ構成

#### A. 士業サイト（例: 司法書士）
1. TOP（実績・専門領域・無料相談 CTA）
2. 業務案内（不動産登記・商業登記・相続・会社設立 など）
3. 料金（明朗会計型）
4. 事務所案内（代表挨拶・所在地）
5. お問い合わせ
6. プライバシーポリシー
7. 特定商取引法表記（必要に応じて）

#### B. 飲食サイト（例: 和食店）
1. TOP（料理写真・コンセプト・予約導線）
2. お品書き / メニュー（ランチ・ディナー）
3. 店内・空間紹介
4. 店主・職人紹介
5. アクセス・営業時間
6. ご予約（フォームまたは電話導線）
7. お知らせ・新着

### 2.5 推奨配置オプション

| 案 | 配置 | メリット | デメリット |
|---|------|---------|---------|
| **案 1** | `tcharton.com/cases/demo-shigyo/` `tcharton.com/cases/demo-inshoku/` | 単一リポジトリで管理、SPEC 検証統合 | URL 構造が tcharton.com 配下、独立性に劣る |
| **案 2** | `demo1.tcharton.com` `demo2.tcharton.com` | 独立性あり、本物感 | DNS 設定・別リポジトリ管理が必要 |
| **案 3** | 別ドメイン（架空・取得） | 完全独立 | 費用発生、運用負荷 |

**推奨**: **案 1** で着手 → 実用感が増したら案 2 へ移行検討。

### 2.6 ビルド順序（推奨）

1. **業種選定 + 設計仕様確定**（ユーザー合意）
2. **デザイントークン策定**（業種固有カラー・タイポ・写真トーン）
3. **A: 士業サイト**を先に構築（情報構造が単純で型化しやすい）
4. **A: spec-checker S-RANK 達成 → 本番展開**
5. **B: 飲食サイト**を構築（写真重視・体験訴求のため難度高め）
6. **B: spec-checker S-RANK 達成 → 本番展開**
7. **`cases/index.html` を実物リンク掲載に更新**
8. **TOP「導入事例」セクションも実物展示に変更**

### 2.7 spec-checker.js への影響

  // ...

各デモサイトの `getVariant()` マッピングも追加。

## 2C. FAQ 拡充プロジェクト（提案③統合 / **優先順位 1**）

### 2C.1 目的・根拠
- **GEO 論文（Aggarwal et al., KDD 2024）**で実証された **Fluency Optimization** により、対話型 AI（ChatGPT / Perplexity / Claude 等）からの引用率を最大化
- E-E-A-T への直接寄与（経験・専門性・権威性・信頼性）
- 既存 `/faq/` の **FAQPage JSON-LD** を活用拡張するため、**インフラ追加コストゼロ**

### 2C.2 設計方針

| 項目 | 内容 |
|------|------|
| **質問形式** | 自然言語そのまま（例：「保守運用を怠るとどうなる？」「個人事業主でも AI 予測は導入できる？」） |
| **回答構造** | 結論 1 文 → 箇条書き理由 → 一次ソース URL |
| **統計引用** | IPA / JNSA / 警察庁 / 中小企業庁 / 経済センサス等の**公的一次ソースのみ**。捏造禁止（SPEC §0.0 H-1） |
| **カテゴリ拡張** | 既存 3 カテゴリ（WEB/保守/AI予測）に加え、新カテゴリ「料金」「契約・納期」「セキュリティ」「個人事業主向け」「地域・対応エリア」検討 |

### 2C.3 規模・工数（2026-04-25 実態反映）

| 段階 | 質問数 | 期間 | 状態 |
|:---:|:---:|:---:|:---:|
| 現状 | **40 問**（4 カテゴリ：WEB 15 / 保守 10 / AI予測 10 / 共通 5） | — | ✅ Phase 1 完了済 |
| Phase 2 目標 | **50〜60 問**（+10〜20 問・**質的拡充**重視） | 2〜4 週間 | 🟢 着手中 |
| Phase 2 重点 | **公的統計引用付き** + **LLM クエリ最適化文体**の追加 | — | — |

### 2C.4 Phase 2 拡充方針（**質的アップグレード**）
- 既存 40 問は維持（数の追加より**統計裏付け強化**を優先）
- 新規追加は **対話 AI 想定クエリ**形式：「○○するとどうなる？」「○○の3つのステップは？」
- 追加質問には**公的一次ソースの統計を最低 1 件**埋込（IPA / 中小企業庁 / 経済センサス / OWASP / 警察庁等）
- 新カテゴリ候補：「セキュリティ」「個人事業主向け」「地域・対応エリア」

### 2C.4 spec-checker への影響
- `/faq/index.html` のみの更新で完結（STATIC_TARGETS 追加なし）
- FAQPage JSON-LD は質問追加に応じて自動拡張
